# PSOP 产品安全规划 v1

## 1. 文档定位

本文从产品规划角度定义 PSOP 如何保证安全。

这里的安全不是单点的敏感词过滤，也不是把模型回答限制得更保守，而是回答一个更核心的问题：

> 当 PSOP 帮助用户执行现实世界任务时，产品如何让任务边界清晰、步骤可信、输出可控、风险可停、责任可追溯。

本文用于指导后续产品设计、交互设计、Skill 设计规范、编译诊断、运行体验、审计回放与企业治理能力的规划。它不替代以下事实源：

- [产品愿景](vision.md)
- [系统架构设计](../architecture/system-architecture.md)
- [Execution Graph 形式定义](../architecture/execution-graph-formal-v5.md)

## 2. 产品安全主张

PSOP 的安全主张是：

> 让 AI 参与现实世界任务，但不让 AI 独占任务判断、现场授权、状态提交和责任解释。

因此，PSOP 的产品安全由五个原则组成：

1. **先定义边界，再执行任务**
   - 每个 Skill 都必须让用户知道适用条件、不适用条件、准备事项、风险提醒和停止条件。
   - 用户不是在和一个通用聊天机器人对话，而是在启动一个有边界的现实世界任务协作流程。

2. **先验证证据，再推进状态**
   - PSOP 不应因为用户一句“完成了”就默认任务完成。
   - 对需要现场确认、图片、文件、传感器、设备回执或审批的步骤，产品必须鼓励或要求用户提供证据。

3. **高风险动作必须可停、可转人工、可升级**
   - 涉及人身安全、设备安全、生产系统写入、财务影响、工单关闭、合规判断的动作，不能只依赖模型自行判断。
   - 产品需要提供停止、转人工、请求专家、请求主管确认等路径。

4. **模型输出只是建议，不是最终授权**
   - LLM 可以生成指令、解释、判断建议和总结。
   - 但现场授权、企业审批、正式状态提交和高风险能力调用必须有产品边界。

5. **所有关键安全过程必须可回放**
   - 事后需要能回答：谁发起了任务，Skill 当时的安全边界是什么，模型说了什么，用户提交了什么证据，系统为什么继续、停止或升级。

## 3. PSOP 需要保护什么

### 3.1 保护现场用户

现场用户可能处在设备、产线、能源、医疗、巡检、维修等复杂环境中。PSOP 需要避免：

- 给出危险、过度自信或缺少前置确认的操作建议。
- 在用户缺少资质、工具、防护或现场条件不满足时继续推进任务。
- 把模型判断包装成确定结论，导致用户忽略企业规程或现场负责人。
- 在异常、危险、证据不足时仍然鼓励用户完成任务。

### 3.2 保护企业系统

PSOP 可能连接工单、资产、知识库、MCP 工具、设备、传感器、对象存储和外部系统。产品需要避免：

- 让用户或模型越权读取、写入或关闭业务对象。
- 让未经确认的输出进入生产系统。
- 让外部工具调用绕过审计、预算、审批和回放。
- 让一个 Skill 的风险扩散到其他系统或其他运行实例。

### 3.3 保护数据和隐私

PSOP 会处理现场照片、视频、日志、用户描述、设备读数、企业知识和模型输入输出。产品需要避免：

- 隐私、密钥、凭证、客户信息、设备敏感参数被无意暴露。
- Replay 或调试页面把敏感数据默认展示给不该看到的人。
- 用户上传的文件被当成可信内容直接传播给模型、工具或外部系统。
- 为了排障保留过多不必要的原始数据。

### 3.4 保护 Skill 生态

PSOP 的长期价值来自可复用 Skill。产品需要避免：

- 低质量 Skill 被发布并用于真实任务。
- Skill 没有写清楚适用范围、安全约束、证据要求和失败路径。
- Skill 作者把危险动作、越权动作或不合规建议包装成普通步骤。
- Skill 演进时丢失原有安全边界。

### 3.5 保护模型使用边界

PSOP 使用大模型，但不能让大模型成为无限制入口。产品需要避免：

- 生成色情、暴力、仇恨、违法、极端主义、自伤等不安全内容。
- 生成政治劝服、敏感群体攻击或其他不适合企业生产场景的内容。
- 在医疗、法律、金融、安全生产等高风险领域给出超出产品定位的确定性结论。
- 被用户通过 prompt injection 诱导绕过系统边界、工具策略或安全提醒。

## 4. 安全贯穿的产品旅程

### 4.1 创建 Skill 时：让安全成为任务设计的一部分

Skill Studio 不应只让用户写目标和步骤，还应引导用户回答：

- 这个 Skill 适合什么场景，不适合什么场景？
- 执行前用户必须具备什么工具、资质、材料或环境条件？
- 哪些情况必须停止？
- 哪些情况必须找专家、主管、现场负责人或企业系统确认？
- 每一步需要什么证据才算完成？
- 最终完成标准是什么？
- 如果证据不足、用户中断、现场危险或任务不适用，应该怎么退出？

产品体验上，安全不应藏在高级配置里，而应成为 Skill 创建和发布前的自然检查项。

### 4.2 发布 Skill 时：把安全作为质量门槛

发布不是简单保存版本，而是把一个 Skill 从草稿变成可运行契约。发布阶段应提供安全审查体验：

- 展示 Skill 的安全完整性摘要。
- 标出缺失的适用边界、停止条件、证据要求和恢复路径。
- 对高风险能力给出明显提示。
- 对缺少人工确认、审批或转人工路径的危险流程给出阻断或强提醒。
- 让发布者知道这个 Skill 将以什么风险等级进入运行环境。

### 4.3 发起运行时：让用户知道自己正在进入什么任务

用户启动 Skill 时，PSOP 应先建立安全预期：

- 当前任务摘要。
- 适用范围和不适用范围。
- 开始前准备事项。
- 关键风险提醒。
- 用户需要承担的现场确认责任。
- 需要准备的证据类型。

这一步的产品目标不是制造冗长免责声明，而是让用户在进入任务前形成正确心智：PSOP 是协作执行工具，不是现场负责人替代品。

### 4.4 执行过程中：让每一步都可判断、可暂停、可恢复

运行时的安全体验应围绕现实步骤组织：

- 每一步只给当前需要执行的指令，不把后续复杂计划一次性压给用户。
- 每一步展示和当前动作相关的安全提醒。
- 用户提交证据后，系统再判断是否可推进。
- 证据不足时，产品应要求补充，而不是勉强通过。
- 出现危险、异常、不适用或用户不确定时，应提供停止、重试、转人工和升级路径。

产品上要避免“为了完成流程而完成流程”。安全的 PSOP 应该允许任务正确失败。

### 4.5 能力调用时：让用户理解 AI 正在调用什么

当 PSOP 需要调用模型、MCP 工具、企业系统、外部 Skill、代码执行或设备能力时，产品需要让风险可见：

- 这是只读查询，还是会写入系统？
- 会不会影响工单、资产、设备、账务或生产状态？
- 是否需要用户确认或审批？
- 调用结果是建议，还是会成为后续步骤依据？
- 调用失败后是否可以重试、跳过或转人工？

对于高风险能力，产品应默认使用更保守的体验：确认、审批、隔离、可撤销说明和完整审计。

### 4.6 完成任务时：完成必须经过最终确认

PSOP 的完成不应等同于“模型生成了最后一句话”。产品应要求最终确认：

- 关键步骤是否都已完成。
- 必要证据是否齐全。
- 是否触发过停止条件或未解决异常。
- 是否还有需要人工确认的高风险结果。
- 最终输出是否适合写入用户、企业系统或工单记录。

只有通过最终确认，任务才应展示为成功完成。

### 4.7 回放复盘时：让安全过程可解释

Replay 不只是排障工具，也应是安全复盘工具。它需要让产品、实施、客户和审计人员看清：

- 任务启动时的安全边界。
- 每一步的用户输入、系统输出和证据。
- 模型是否输出过敏感或高风险内容。
- 系统是否进行过提醒、阻断、脱敏、转人工或升级。
- 高风险能力是否经过确认。
- 任务为什么成功、失败、取消或升级。

## 5. 产品安全能力地图

### 5.1 Safety Contract：Skill 的安全契约

每个 Skill 都应具备面向用户和企业的安全契约，至少表达：

- 适用范围。
- 不适用范围。
- 执行前条件。
- 现场风险。
- 禁止操作。
- 停止条件。
- 证据要求。
- 转人工条件。
- 完成标准。

这是 PSOP 产品安全的起点。

### 5.2 Safety Review：发布前安全检查

发布前安全检查帮助 Skill 作者发现缺口：

- 这个 Skill 是否缺少风险说明？
- 是否有步骤没有证据要求？
- 是否有危险动作缺少停止条件？
- 是否有高风险能力缺少确认路径？
- 是否存在“用户说完成就完成”的薄弱环节？

### 5.3 Safety Mode：运行风险模式

不同 Skill 和不同运行环境应有不同安全模式，例如：

- `normal`：普通指导任务。
- `evidence_required`：关键步骤必须提交证据。
- `supervised`：运行中需要人工监督或专家参与。
- `approval_required`：关键动作必须审批。
- `restricted`：只能提供信息和建议，不允许写入系统或控制设备。

这些模式是产品心智，不要求一次性实现完整策略系统。

### 5.4 Sensitive Content Guard：敏感内容保护

PSOP 应把敏感内容保护纳入用户体验：

- 用户输入危险、违法、色情、暴力、自伤、仇恨、极端主义或不适合企业场景的请求时，产品应拒绝、降级或引导到安全替代方案。
- 模型输出包含敏感内容时，产品应阻断、脱敏、摘要化或要求重新生成。
- Replay 展示敏感内容时，默认应降低暴露面。

### 5.5 Evidence Gate：证据门槛

证据门槛让 PSOP 避免虚假完成：

- 关键步骤要求文本、图片、视频、文件、设备 ACK、传感器读数或人工确认。
- 证据不足时，产品明确告诉用户缺什么。
- 证据冲突时，产品不应继续推进到成功。
- 最终完成需要回看关键证据。

### 5.6 Human Handoff：人工接管

PSOP 应把转人工做成正常路径，而不是异常失败：

- 用户不确定时可以请求人工。
- 系统识别危险、不适用或证据冲突时可以建议人工。
- 高风险能力执行前可以要求主管、专家或现场负责人确认。
- 人工接管本身也应被记录到回放中。

### 5.7 Safe Capability Boundary：能力边界

PSOP 应明确哪些事情只能建议，哪些事情可以调用工具，哪些事情必须确认：

- 信息查询通常低风险。
- 业务系统写入需要更高确认。
- 设备控制、工单关闭、财务影响、人员安全相关动作属于高风险。
- 外部 Skill 和 MCP 工具不应因为接入方便就自动获得高权限。

### 5.8 Safety Replay：安全回放

安全回放应帮助客户建立信任：

- 运行前边界是什么。
- 运行中提示了哪些风险。
- 哪些输入或输出被拦截、脱敏或升级。
- 哪些证据支撑了完成判断。
- 哪些人、工具或模型参与了关键决策。

## 6. 安全决策的产品语义

PSOP 面对风险时，不应只有“通过”和“失败”两种结果。产品层可以采用以下决策语义：

| 决策 | 产品含义 |
| --- | --- |
| `allow` | 风险可接受，继续执行。 |
| `warn` | 可以继续，但需要展示风险提醒或补充注意事项。 |
| `redact` | 内容可保留为事实，但展示或传给下游前需要脱敏、摘要或隐藏。 |
| `block` | 当前输入、输出或能力调用不允许继续。 |
| `require_approval` | 需要用户、主管、专家或企业流程确认后才能继续。 |
| `abort` | 当前任务不适合继续，应安全终止。 |
| `escalate` | 需要转人工、转专家、转主管或进入企业应急流程。 |

这些决策应体现在用户体验中，让用户知道系统为什么停、为什么提醒、为什么需要更多证据或人工确认。

## 7. 分阶段产品路线图

### 7.1 v1：建立安全心智和基础闭环

v1 重点是让安全进入主链路：

- Skill 创建时必须表达安全约束、停止条件和证据要求。
- Skill 发布前展示安全完整性检查。
- Run Live 展示任务摘要、适用边界、安全提醒和准备事项。
- 关键步骤要求证据后再推进。
- 敏感内容和明显危险请求需要拒绝、提醒或降级。
- Replay 能看到安全提醒、证据、模型输出和最终完成判断。

### 7.2 v2：建立企业级安全治理体验

v2 重点是让企业可以治理：

- Skill 风险等级。
- 安全策略模板。
- 行业安全包。
- 高风险能力审批。
- 数据敏感等级和脱敏视图。
- 安全事件检索和统计。
- 安全 Dashboard。

### 7.3 v3：建立可认证、可审计的安全体系

v3 重点是让 PSOP 支持更严格行业：

- 高风险 Skill 的形式化检查。
- 企业合规策略包。
- 安全事件工单化。
- 审计导出。
- 外部 EHS、质量、合规系统联动。
- 关键流程的证明型回放和责任边界说明。

## 8. 产品验收指标

后续可以用以下指标判断 PSOP 安全能力是否有效：

- 发布的 Skill 中，具备安全约束和停止条件的比例。
- 关键步骤中，配置证据要求的比例。
- 因证据不足而正确阻止推进的次数。
- 高风险能力调用前触发确认或审批的比例。
- 敏感内容被拒绝、脱敏或降级的次数。
- 安全事件在 Replay 中可完整复盘的比例。
- 用户主动转人工或系统建议转人工的成功处理比例。
- 客户安全审计中能解释清楚的运行样本比例。

## 9. 非目标

当前阶段不把 PSOP 定义为完整的企业权限中心、EHS 系统、合规审批平台或内容审核平台。

PSOP 的产品安全目标是：

- 让 Skill 本身具备安全边界。
- 让运行过程不会盲目推进。
- 让高风险能力不会绕过确认。
- 让模型输出不会成为不可追责的黑箱。
- 让安全事件能够被看见、解释和复盘。

PSOP 不承诺替代现场负责人、企业规程、法律合规判断、专业资质审核或真实物理安全系统。

## 10. 一句话总结

PSOP 保证安全的方式，不是把 AI 变成一个更谨慎的聊天机器人，而是把现实世界任务拆成有边界、有证据、有停止条件、有人工接管、有回放审计的产品流程，让智能始终工作在可控、可解释、可追责的任务框架内。
